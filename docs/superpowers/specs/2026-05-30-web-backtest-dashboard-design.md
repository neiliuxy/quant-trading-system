# Web 回测分析台设计

## 目标

把现有量化回测项目做成一个可本地运行、后续可部署到服务器的 Web 分析台。第一版以单次回测分析为主，支持异步任务、历史记录复用、市场过滤结果查看、前后端分离界面，并能在电脑和手机上访问。

## 范围

### 进入首版

- 单次回测任务提交
- 异步任务执行与状态轮询
- 历史任务保留与结果复用
- 回测结果详情页
- 市场评分分解展示
- 无登录的单用户本地访问
- 响应式前端，兼容桌面和手机

### 暂不进入首版

- 用户登录、权限、账号体系
- 多人协作与分享链接
- 实时推送、WebSocket
- 批量选股主流程
- 独立 worker 集群

## 总体架构

### 后端

- `server/`：FastAPI 应用
- `server/db.py`：SQLite 连接与迁移
- `server/jobs.py`：任务创建、查重、状态流转
- `server/api.py`：HTTP 接口
- `server/executor.py`：后台线程执行任务

### 前端

- `web/`：React + Vite
- `web/src/pages/Workbench.tsx`：研究工作台
- `web/src/pages/JobDetail.tsx`：任务详情
- `web/src/components/`：表单、图表、表格、状态徽标

### 现有核心逻辑

复用现有模块，不拆散核心业务：

- `backtest/`：回测入口与数据加载
- `market/`：市场评分
- `strategies/`：策略实现

建议新增一个轻量服务层，例如 `backtest/service.py`，把 CLI 型流程整理为可被 Web 调用的函数，避免直接复用打印输出逻辑。

## 关键设计

### 任务模型

任务是首版系统的核心实体。每次提交回测都会形成一个任务记录，包含：

- 输入参数：`symbol`、`start`、`end`、`cash`、`use_market_filter`
- 策略参数：`risk_percent`、均线参数、市场配置
- 代码版本：`git commit hash` 或等价版本号
- 去重键：`run_key`
- 状态：`queued`、`running`、`completed`、`failed`、`cancelled`
- 摘要结果：收益率、最大回撤、交易次数、胜率、期末资金

`run_key` 由“输入参数 + 策略配置 + 市场配置 hash + 代码版本”共同决定。相同 `run_key` 且已有成功结果时，直接复用历史结果，不重新执行。

### 任务执行

第一版采用单进程后台线程执行：

1. API 收到提交请求
2. 计算 `run_key`
3. 命中历史成功结果则直接返回对应任务
4. 未命中则入库为 `queued`
5. 后台线程更新为 `running`
6. 调用现有回测与市场评分逻辑
7. 保存摘要、明细、错误信息
8. 任务完成后置为 `completed` 或 `failed`

这个实现足够覆盖首版需求，并保留后续替换为独立 worker 的空间。

## 数据存储

### SQLite

SQLite 只保存结构化索引数据：

- `jobs`：任务主表
- `job_params`：任务参数快照
- `job_results`：摘要指标
- `job_artifacts`：文件型结果索引

### 文件缓存

较大的结果明细不直接塞进数据库，保存到：

- `data/results/{job_id}.json`

建议内容包括：

- 资金曲线
- 交易明细
- 持仓变化
- 市场评分序列
- 过滤器对比摘要

行情数据与市场评分缓存继续沿用 `data/` 下现有 CSV 机制。

## API 设计

### 提交任务

- `POST /api/jobs`
- 输入：回测参数与策略参数
- 输出：`job_id`、`run_key`、状态

### 查询任务

- `GET /api/jobs`
- 返回最近任务列表，可筛选状态、标的、日期范围

### 查询任务详情

- `GET /api/jobs/{job_id}`
- 返回任务参数、状态、摘要、缓存命中信息

### 查询任务结果

- `GET /api/jobs/{job_id}/result`
- 返回图表数据、交易明细、评分分解、对比视图所需数据

### 强制重跑

- `POST /api/jobs/{job_id}/rerun`
- 忽略已有成功结果，创建新任务

## 前端布局

### 桌面端

首页采用左右工作台布局：

- 左侧：参数表单、提交按钮、历史任务列表、缓存命中提示
- 右侧：当前任务或最近结果详情

结果区从上到下分为：

- KPI 条：收益率、最大回撤、胜率、交易次数、期末资金、市场评分均值/范围
- 主图：资金曲线、交易点、基准线
- 市场评分：趋势 / 情绪 / 量能 / 总分分解
- 明细表：交易列表、参数快照、缓存信息
- 对比视图：有过滤 vs 无过滤

### 移动端

移动端采用单列堆叠：

- 表单优先
- KPI 卡片
- 图表区
- 明细折叠区
- 历史任务抽屉或底部列表

核心目标是信息不丢失，而不是把桌面布局硬缩小。

## 结果复用

“测过的不用重新跑”按以下规则判定：

- 同 `symbol`、`start`、`end`、`cash`
- 同 `use_market_filter`
- 同策略参数
- 同市场配置 hash
- 同代码版本

命中时直接返回历史任务结果，并在前端显示“缓存命中”。用户可通过“强制重跑”创建新任务。

## 错误处理

- 数据加载失败：任务标记为 `failed`，记录错误原因
- 市场评分失败：如果启用过滤器，任务失败并保留错误；不静默退回成不带过滤的结果
- 回测异常：任务失败，前端展示错误摘要
- 前端轮询失败：提示任务状态未知，但不打断历史浏览

首版不做登录，所以不加权限层；部署到服务器时应放在内网或反向代理之后。

## 测试

### 后端

- 任务提交与查重
- 同参数结果复用
- 强制重跑生成新任务
- 状态流转正确
- 失败任务能记录错误信息

### 核心逻辑

- 回测结果可序列化
- 市场评分数据可被前端消费
- 代码版本变化会让缓存失效
- 手机与桌面共用同一 API

### 前端

- 参数提交后能创建任务
- 任务状态能轮询更新
- 历史任务可切换
- 对比视图能展示有/无过滤差异
- 移动端布局不丢关键内容

## 实现顺序

1. 后端任务表与 SQLite
2. 回测服务函数化
3. FastAPI 任务 API
4. React/Vite 工作台页面
5. 结果图表与详情页
6. 历史复用与强制重跑
7. 响应式收尾与测试
