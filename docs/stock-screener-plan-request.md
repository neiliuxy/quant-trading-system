### 项目背景

这是一个个人量化交易系统，面向 A 股日线低频交易，目标交易频率为每月 2-4 次。当前系统已经具备以下基础能力：

- 策略注册与回测
- 市场过滤评分
- DataHub 数据缓存
- FastAPI 后端任务系统
- React/Vite Web 分析台
- Walk-forward Optimization

仓库大致结构如下：

- `strategies/`：策略实现与注册表
- `backtest/`：回测编排、CLI 入口、指标、成本、Walk-forward
- `datahub/`：统一数据门面、缓存、刷新任务、元数据
- `market/`：市场评分系统
- `server/`：FastAPI 后端
- `web/`：React + Vite + TypeScript 前端
- `tests/`：pytest 测试

系统当前已有一些相关能力可参考：

- 批量选股回测入口：`backtest/stock_selector.py`
- 市场过滤设计文档：`docs/superpowers/specs/2026-05-30-market-filter-design.md`
- 已注册策略总览：`docs/architecture.md`

### 需求目标

我要新增一个“股票初筛（Stock Screener）”能力，用于在 A 股全市场或指定股票池中，先把几千只股票筛成几十只候选，再交给已有策略做最终入场判断。

这个模块不是直接生成买卖信号，而是做“前置漏斗式筛选”。

目标是：

- 把 `5000+` 只 A 股缩小到 `20-100` 只候选
- 适配个人低频量化，避免高换手和过度交易
- 筛出来的股票更适合现有低频趋势或波段策略
- 支持后续回测验证“初筛 + 策略”的组合效果

### 核心业务思路

股票初筛采用 6 层漏斗：

1. 市场环境过滤
2. 基础股票池限定
3. 风险与可交易性过滤
4. 趋势过滤
5. 强弱评分排序
6. 输出候选池给策略或回测模块继续使用

#### 1. 市场环境过滤

目标：只有在市场环境允许时，才产生候选池，避免熊市乱开仓。

建议逻辑：

- 使用现有 `market/` 模块的市场评分能力
- 当市场评分低于某阈值时：
  - 可以直接不输出候选
  - 或者仅输出少量高分股票
- 当市场评分较高时：
  - 正常执行全量初筛

需要在 Plan 中明确：

- 是将市场过滤作为 screener 前置硬门槛
- 还是作为最终排序加权项
- 或者两者兼有

#### 2. 基础股票池限定

目标：不要默认从全市场裸跑，而是支持可配置股票池。

至少支持以下股票池来源：

- 全 A
- 沪深 300
- 中证 500
- 自定义股票列表
- 行业或主题成分股列表，后续可扩展

需要支持两种模式：

- `universe_mode = full_market`
- `universe_mode = predefined/custom`

注意：

- 默认应优先支持 `沪深300`、`中证500`、`自定义列表`
- 不要求第一版就把所有指数成分和行业成分都做完，但架构要可扩展

#### 3. 风险与可交易性过滤

目标：剔除不适合个人低频交易的股票。

第一版建议支持以下过滤条件：

- 剔除 `ST/*ST`
- 剔除上市时间不足 `N` 个交易日的次新股
- 剔除长期停牌或近期无有效行情数据的股票
- 剔除近 `M` 日日均成交额低于阈值的股票
- 可选：剔除极端波动、连续一字板、连续涨停后高位加速的股票

这里需要 Plan 里回答：

- 第一版哪些规则必须做
- 哪些作为可选扩展
- 这些规则的数据来源放在哪一层取数，是 `datahub/` 还是直接在筛选模块中处理

#### 4. 趋势过滤

目标：保留中期趋势较好的股票，避免把大量下跌股交给策略继续误筛。

第一版建议至少支持这些趋势条件：

- `close > ma20`
- `ma20 > ma60`
- `ma60` 最近若干日斜率为正
- 近 `60` 日收益率大于基准指数同期收益率

需要设计为：

- 支持规则组合
- 支持参数化阈值
- 支持后续新增别的趋势条件

#### 5. 强弱评分排序

目标：对通过过滤的股票继续评分排序，而不是只做硬筛选。

建议评分维度：

- 相对强度评分：近 `20/60` 日跑赢指数的程度
- 趋势质量评分：均线多头程度、趋势平滑程度
- 回撤质量评分：近阶段回撤大小
- 量价配合评分：上涨放量、调整缩量
- 流动性评分：成交额、成交连续性

要求：

- 支持总分 = 各子分项加权求和
- 权重可配置
- 输出排序结果
- 可截取 `top_n`

需要在 Plan 中说明：

- 评分引擎是函数式实现还是类实现
- 如何保证后续可新增维度
- 评分结果如何落库或缓存

#### 6. 输出候选池

输出结果至少包括：

- 交易日
- 股票代码
- 股票名称
- 所属股票池
- 是否通过各层过滤
- 各项评分
- 总分
- 排名
- 市场评分
- 关键解释字段，例如为什么入选

希望最终候选池可以被以下模块消费：

- 回测模块
- 策略模块
- Web 前端展示
- 后端 API 查询

### 第一版范围

第一版只做日线低频，不做分钟级，不做实盘下单。

第一版优先支持：

- 输入一个日期或日期区间
- 输入一个股票池
- 执行初筛
- 输出候选列表和评分
- 支持把候选列表交给后续回测使用
- 支持 CLI 或服务层调用
- 支持后端 API 查询结果

第一版可以暂不做：

- 复杂基本面因子
- 舆情、新闻、NLP
- 财报质量评分
- 板块轮动联动打分
- 实时盘中筛选
- 自动下单

### 功能需求

请在 Plan 中覆盖以下功能模块设计。

#### A. Screener 核心服务

需要新增一个核心服务层，例如：

- `backtest/stock_screener.py`
- 或 `screening/service.py`

职责包括：

- 加载股票池
- 拉取行情
- 执行多层过滤
- 计算评分
- 输出候选池结果

#### B. 配置模型

需要有结构化配置，而不是到处散落参数。

建议包含：

- 股票池配置
- 过滤条件配置
- 评分权重配置
- TopN 配置
- 市场过滤配置

请考虑是否新增类似以下对象：

- `ScreenerRequest`
- `ScreenerFilterConfig`
- `ScreenerScoreConfig`
- `ScreenerResult`

#### C. 数据访问

请设计如何复用现有 `datahub/`：

- 如何批量取多只股票的日线数据
- 如何处理缺失数据
- 如何减少重复加载
- 是否需要对股票池成分做缓存
- 是否需要支持分批处理，避免一次性跑全市场过慢

#### D. 回测集成

我希望后续支持两种回测方式：

- 先初筛，再只对候选股票运行策略回测
- 对比“有初筛”和“无初筛”的效果差异

Plan 里需要说明：

- 初筛结果如何传递给回测服务
- 是在回测前先生成候选池，还是边回测边筛选
- 如何设计实验对照，验证初筛模块是否真的提升结果

#### E. 后端 API

希望至少有这些接口方向：

- 提交一次初筛任务
- 查询初筛结果
- 查询某日候选池
- 查询某只股票的入选原因或评分明细

请给出建议 API 设计，不要求完全定死实现细节，但需要有清晰的资源模型。

#### F. 前端展示

第一版前端可以简单，但希望可展示：

- 筛选参数
- 候选股票列表
- 排名与评分
- 入选原因说明
- 可跳转到回测

前端不需要在第一版做得很复杂，但 Plan 中要说明最小可用界面是什么。

#### G. 测试

需要覆盖：

- 过滤规则单测
- 评分逻辑单测
- 服务层集成测试
- API 测试
- 回测联动验证

要重点避免：

- 未来函数
- 使用未来数据泄漏
- 指标计算窗口不一致
- 停牌或缺失数据导致筛选异常

### 非功能要求

- 代码风格与现有仓库保持一致
- Python 3，4 空格缩进，命名使用 `snake_case`
- 优先复用现有模块，不重复造轮子
- 第一版要可维护、可扩展，不要写死在单一策略里
- 筛选逻辑与策略逻辑要分层，不要耦合死
- 要考虑全市场运行性能，但第一版不必过度优化
- 需要考虑结果缓存与可复现性
- 所有结果都必须基于“当时可见数据”，避免未来函数

### 业务规则建议

请把下面这些规则作为第一版优先候选，并在 Plan 中标注哪些是 MVP，哪些是扩展项。

#### MVP 过滤规则

- 非 ST
- 上市超过 180 天
- 近 20 日日均成交额 > 1 亿
- 近 60 日日线数据完整度满足最低要求
- `close > ma20`
- `ma20 > ma60`
- `ma60` 斜率向上
- 近 60 日收益强于沪深300同期收益

#### MVP 评分规则

- 相对强度
- 趋势质量
- 回撤控制
- 量价配合
- 流动性

#### MVP 输出

- TopN 候选池
- 各项分数
- 总分排名
- 入选原因摘要
- 市场评分
